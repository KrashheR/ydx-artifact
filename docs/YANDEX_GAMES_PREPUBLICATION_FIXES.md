# Предпубликационные фиксы для Яндекс Игр

## Проект
Игра в жанре «Найди отличия» с тремя кампаниями по 13 уровней.

## Цель задачи
Подготовить проект к публикации в Яндекс Играх и устранить все найденные технические, контентные и UX-проблемы.

Главный результат работы: production-сборка, которая корректно запускается в Яндекс Играх, не содержит незавершённых функций, проходит мобильные сценарии, корректно использует SDK и упакована в валидный ZIP.

---

# Общие правила работы

1. Сначала изучить структуру исходного проекта и определить реальные файлы и компоненты, отвечающие за перечисленные ниже функции.
2. Не переписывать игру целиком и не менять существующую визуальную концепцию без необходимости.
3. Не удалять рабочую интеграцию Yandex Games SDK.
4. Не ломать существующие сохранения пользователей.
5. Все dev-only инструменты должны быть исключены из production bundle.
6. После каждого крупного изменения запускать сборку и тесты.
7. В конце предоставить:
   - список изменённых файлов;
   - краткое описание каждого исправления;
   - результаты тестов;
   - известные оставшиеся ограничения;
   - готовый production ZIP для Яндекс Игр.

# P0 — критические блокеры публикации

## 1. Исправить упаковку production ZIP

### Проблема
Сейчас `index.html` находится внутри папки `dist`, а не в корне ZIP. Также в архиве присутствуют служебные файлы macOS.

Неверно:

```text
archive.zip
├── dist/
│   ├── index.html
│   └── assets/
└── __MACOSX/
```

Должно быть:

```text
archive.zip
├── index.html
└── assets/
```

### Что сделать

- Добавить production-скрипт упаковки.
- Архивировать содержимое `dist`, а не саму папку `dist`.
- Исключить:
  - `.DS_Store`;
  - `__MACOSX`;
  - файлы `._*`;
  - source maps, если они не нужны в production;
  - другие системные и временные файлы.
- Проверять наличие `index.html` в корне архива автоматически.

### Рекомендуемый npm-скрипт

Можно реализовать Node.js-скрипт `scripts/package-yandex.mjs`, который:

1. удаляет старый архив;
2. проверяет наличие `dist/index.html`;
3. создаёт ZIP из содержимого `dist`;
4. проверяет содержимое ZIP;
5. завершает процесс с ошибкой, если `index.html` не в корне.

### Критерии приёмки

- В корне ZIP находится `index.html`.
- В ZIP отсутствует папка `dist`.
- В ZIP отсутствуют `__MACOSX`, `.DS_Store`, `._*`.
- Игра запускается из итогового ZIP в черновике Яндекс Игр.

- Проверить уровень вручную без подсказок.
- Проверить уровень с подсказками.

### Важно

- Не делать отличия микроскопическими.
- Не размещать hotspot в пустом месте.
- Не создавать перекрывающиеся hotspot.
- Каждое отличие должно однозначно определяться на изображении.

### Критерии приёмки

- Файлы A и B не идентичны.
- На уровне действительно присутствует ровно 9 отличий.
- Все 9 отличий можно найти вручную.

## 3. Удалить незавершённый paywall из первой production-версии

### Проблема
В интерфейсе присутствует покупка «Открыть все кампании», но Payments API фактически не подключён:

- цена выводится некорректно;
- покупка не запускается;
- восстановление покупок не работает;
- отсутствует полноценная обработка покупок;
- присутствуют обещания платного доступа, которые невозможно выполнить.

### Решение для первой публикации

Полностью удалить или скрыть paywall до отдельного обновления.

### Что удалить из production UI

- кнопку «Открыть все экспедиции»;
- кнопки покупки на заблокированных кампаниях;
- paywall-модалку;
- кнопку «Восстановить покупки»;
- отображение цены;
- обещание доступа ко всем будущим кампаниям;
- dev-покупку `dev-all-campaigns`;
- неиспользуемые purchase-флаги и mock-данные.

### Что оставить

- последовательное бесплатное открытие кампаний;
- открытие следующей кампании после завершения предыдущей;
- понятное описание условия разблокировки.

### Важно

Не оставлять неактивные кнопки покупки, цену `—`, заглушки и элементы «скоро» внутри production-потока.

### Критерии приёмки

- В production-интерфейсе нет неработающих покупок.
- Пользователь понимает, как открыть следующую кампанию.
- Все три кампании можно открыть игровым способом.
- В bundle отсутствуют mock-покупки и dev-разблокировка.

## 4. Добавить автоматическое определение языка через Yandex SDK

### Проблема
Игра всегда стартует на русском и не учитывает:

```ts
ysdk.environment.i18n.lang
```

### Требуемая логика

При первом запуске:

1. Получить язык окружения из Yandex Games SDK.
2. Преобразовать его в поддерживаемый язык игры.
3. Установить язык интерфейса.
4. Сохранить выбранный язык.

При последующих запусках:

- если пользователь уже вручную выбрал язык, использовать сохранённое значение;
- не перезаписывать ручной выбор языком SDK.

### Рекомендуемое сопоставление

```ts
const russianFallbackLanguages = new Set([
  'ru',
  'be',
  'kk',
  'uk',
  'uz',
]);

function resolveInitialLocale(sdkLanguage?: string): 'ru' | 'en' {
  const normalized = sdkLanguage?.toLowerCase().split('-')[0];
  return normalized && russianFallbackLanguages.has(normalized)
    ? 'ru'
    : 'en';
}
```

### Дополнительно

При смене языка обновлять:

```ts
document.documentElement.lang
```

Также локализовать:

- `<title>`;
- meta description, если она формируется приложением;
- доступные aria-label;
- подписи рекламы и подсказок.

### Критерии приёмки

- Новый пользователь с `environment.i18n.lang = "en"` видит английский интерфейс.
- Новый пользователь с `environment.i18n.lang = "ru"` видит русский интерфейс.
- Ручной выбор языка сохраняется после перезапуска.
- `<html lang>` соответствует текущему языку.
- `<title>` локализован.

## 5. Исправить мобильный HUD игрового уровня

### Проблема
На портретном разрешении около `390 × 844` правая часть HUD выходит за экран:

- кнопки частично обрезаются;
- элементы накладываются;
- часть функций недоступна.

### Что сделать

Переработать мобильную компоновку HUD.

Рекомендуемый состав мобильного HUD:

- назад;
- таймер;
- найдено отличий;
- подсказка;
- пауза или меню.

Второстепенные элементы перенести в меню паузы:

- настройки;
- музыка;
- звук;
- усиленная подсказка;
- выход на карту.

### Требования

- Не уменьшать tap target ниже 44 × 44 CSS px.
- Учитывать safe-area:

```css
padding-left: max(8px, env(safe-area-inset-left));
padding-right: max(8px, env(safe-area-inset-right));
padding-top: max(8px, env(safe-area-inset-top));
```

- Не использовать фиксированную ширину, превышающую viewport.
- Поддержать русский и английский текст без обрезания.

### Критерии приёмки

- На `390 × 844` все обязательные элементы видимы и нажимаемы.
- На `360 × 800` HUD не ломается.
- На `844 × 390` HUD не перекрывает игровую сцену критическим образом.
- Нет горизонтального scroll.
- Нет наложения элементов.

Даже после удаления paywall этот фикс нужен для других модалок:

- подтверждения;

### Что сделать

Создать единый адаптивный контейнер модальных окон.

```css
.modal-panel {
  max-height: calc(100dvh - 24px);

@media (max-height: 500px) and (orientation: landscape) {

    max-height: calc(100dvh - 12px);
```

Учитывать safe-area.

### Критерии приёмки

- Каждую модалку можно полностью просмотреть на `844 × 390`.
- Основные и вторичные кнопки всегда доступны.
- Модалка не выходит за пределы viewport.
- Скролл происходит внутри модалки, а не на body.
- Закрытие доступно без смены ориентации.

### Проблема
После окончания бесплатных подсказок кнопка не сообщает достаточно явно, что будет показана реклама и что пользователь получит за просмотр.

### Что сделать

Когда бесплатные подсказки закончились, заменить обычную кнопку на понятный CTA.

Примеры русского текста:

```text
Смотреть рекламу — получить подсказку
```

Компактный вариант:

```text
Реклама → подсказка
```

Английский вариант:

```text
Watch ad — get a hint
```

Рекомендуется использовать подтверждающую модалку:

```text
Посмотреть короткую рекламу и открыть одно отличие?
```

Кнопки:

- «Смотреть рекламу»;

### Критерии приёмки

- Игрок понимает, что получит одну подсказку.
- Награда не выдаётся при незавершённом просмотре.
- Нет двойной выдачи награды.
- Mobile CTA читается полностью.

# P1 — важные технические исправления

## 8. Починить сохранение и восстановление таймера

### Проблема
В сохранении присутствует `elapsedSeconds`, но значение фактически не обновляется. После обновления страницы таймер может восстановиться полностью.

Также итоговое время может учитывать:

- паузу;
- нахождение приложения в фоне;
- просмотр рекламы;
- системную паузу.

### Что сделать

Создать единый источник истины для активного игрового времени:

```ts
elapsedActiveSeconds
```

Время увеличивается только когда:

- уровень активен;
- игра не на паузе;
- реклама не открыта;
- вкладка видима;
- системная пауза Yandex Games не активна.

### Когда сохранять

- каждые 5–10 секунд;
- после найденного отличия;
- после ошибки;
- при открытии паузы;
- перед рекламой;
- при `visibilitychange`;
- при `pagehide`;
- при выходе на карту;
- при завершении уровня.

### Восстановление

```ts
remainingSeconds = Math.max(
  0,
  LEVEL_DURATION_SECONDS - saved.elapsedActiveSeconds,
);
```

### Миграция

Старые сохранения без актуального значения должны безопасно загружаться:

```ts
const elapsedActiveSeconds = Number.isFinite(saved.elapsedActiveSeconds)
  ? saved.elapsedActiveSeconds
  : 0;
```

### Критерии приёмки

- После перезагрузки оставшееся время не сбрасывается.
- Время рекламы не уменьшает таймер.
- Время паузы не уменьшает таймер.
- Время в фоновой вкладке не уменьшает таймер.
- Старые сохранения не ломаются.

## 9. Удалить production-отладку и внутренние идентификаторы

### Проблема
В интерфейсе видны внутренние подписи вида:

```text
REF-NR-01-SCENE01-A
REF-NR-01-SCENE01-B
```

Также в production bundle присутствуют dev-инструменты и строки редактора hitbox.

### Что сделать

- Удалить внутренние ID из production UI.
- Оставить их только в dev-режиме при необходимости.
- Исключить hitbox editor из production bundle.
- Исключить:
  - `HITBOX EDITOR`;
  - `Apply`;
  - `Copy JSON`;
  - `Reset`;
  - `/__dev/hitboxes/apply`;
  - dev unlock;
  - mock-покупки;
  - тестовые кнопки;
  - debug overlays.

### Требуемый подход

Разнести dev-код по отдельным модулям и подключать только так:

```ts
if (import.meta.env.DEV) {
  const module = await import('./dev/hitbox-editor');
  module.mountHitboxEditor();
}
```

Нельзя просто скрыть dev-компонент CSS-классом. Он не должен попадать в production chunk.

### Критерии приёмки

- На production-экранах нет внутренних ID.
- Поиск по итоговому JS bundle не находит dev endpoint и названия редактора.
- Dev-инструменты по-прежнему можно использовать в development-сборке.

## 10. Добавить legacy browser support или безопасно ограничить браузеры

Статус: реализовано через `@vitejs/plugin-legacy` в `vite.config.ts` для `Safari >= 9`, `iOS >= 9`, `Android >= 5`; production build должен содержать modern bundle, legacy bundle, `nomodule` script и polyfills.

### Проблема
Production bundle использует современные возможности:

- ES modules;
- async/await;
- nullish coalescing;
- arrow functions;
- `Object.fromEntries`;
- `structuredClone`.

Особенно опасен прямой вызов `structuredClone`.

### Что сделать

Минимум:

- убрать обязательную зависимость от `structuredClone`;
- добавить безопасный clone helper;
- настроить подходящий `build.target`;
- рассмотреть `@vitejs/plugin-legacy`;
- добавить необходимые polyfill;
- протестировать Safari и старые Android WebView.

### Безопасный clone helper

Для plain JSON-compatible данных:

```ts
export function cloneJsonValue<T>(value: T): T {
  return JSON.parse(JSON.stringify(value)) as T;
}
```

Если в данных есть `Date`, `Map`, `Set`, `undefined` или классы, реализовать более подходящий вариант.

### Конфигурация Vite

Пример направления, которое нужно адаптировать под текущий проект:

```ts
import legacy from '@vitejs/plugin-legacy';

export default defineConfig({
  plugins: [
    react(),
    legacy({
      targets: ['Safari >= 9', 'iOS >= 9', 'Android >= 5'],
      modernPolyfills: true,
    }),
  ],
  build: {
    target: 'es2018',
  },
});
```

Не копировать конфигурацию без проверки размера и работоспособности bundle.

### Критерии приёмки

- В runtime нет ошибки `structuredClone is not defined`.
- Production build открывается в выбранной минимальной версии Safari.
- Игра корректно стартует в Android WebView, соответствующем выбранной поддержке.
- Размер сборки остаётся разумным.

## 12. Увеличить полезную площадь игровых изображений на desktop

На desktop внутри контейнеров изображений присутствуют большие пустые вертикальные области. Главный игровой контент занимает меньше места, чем мог бы.

```css
aspect-ratio: 1586 / 992;
```

Но проверить реальные размеры всех ассетов и не хардкодить значение, если они различаются.

### Критерии приёмки

- На `1440 × 900` изображения заметно крупнее.
- Все отличия остаются видимыми.
- Нет искажения пропорций.
- Нет лишних огромных пустых полей.

## 13. Улучшить портретный режим сравнения A/B

### Что проверить и улучшить

- Изображение занимает максимально доступную ширину.
- Нет чрезмерных пустых зон сверху и снизу.
- Кнопка переключения A/B легко доступна большим пальцем.
- Переключение не сбрасывает zoom/pan, если они используются.
- Активное изображение A или B обозначено явно.
- Нет случайного двойного засчитывания клика при переключении.
